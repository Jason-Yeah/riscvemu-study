# debug260419：riscvemu 运行 testrv 段错误/无输出排查记录

## 背景

目标：运行本项目的 RISC-V 模拟器 `./riscvemu playground/testrv`，期望输出与 `qemu-riscv64-static playground/testrv` 一致的字符串：

```bash
qemu-riscv64-static playground/testrv
Hello, RISC-V Emulator!
```

最初现象：

```bash
./riscvemu playground/testrv
Segmentation fault (core dumped)
```

以及后续修到不崩溃后，仍然出现“退出码 0 但无输出”的情况。

## 复现与工具

复现命令：

```bash
cd riscvemu
./riscvemu playground/testrv
```

定位手段：

- gdb 获取崩溃地址/回溯
- AddressSanitizer（ASan）定位首次非法访问点
- 对比原作者版本 `/rvemu` 的实现与行为

## Bug 1：解释器分发表（funcs[]）与 enum insn_type_t 不一致导致乱跳

### 现象

早期 gdb 回溯显示在解释执行路径中崩溃，但符号不全，表现为随机地址崩溃/函数指针乱跳。

### 根因

`decode.c` 会解码出 `insn_fence / insn_fence_i` 等类型，但 `interpret.c` 的 `static func_t *funcs[]` 未包含对应槽位，导致：

- 从 `insn_fence` 开始，函数指针表整体错位；
- 之后任何 `insn.type` 索引到错误函数；
- 最终出现非法内存访问或跳转到未知地址触发 SIGSEGV。

另外，CSR 指令在 `enum insn_type_t` 中的顺序为：

```
insn_csrrc, insn_csrrci, insn_csrrs, insn_csrrsi, insn_csrrw, insn_csrrwi
```

但 `funcs[]` 里原先顺序是 `csrrw/csrrs/csrrc/...`，也会进一步造成错位。

### 修复

文件：`riscvemu/src/interpret.c`

- 增加 `func_empty` 作为 `fence/fence_i` 的占位（模拟器中视为 no-op）。
- 调整 `funcs[]` 顺序与 `enum insn_type_t` 严格对齐。
- 增加编译期校验，防止未来再错位：

```c
_Static_assert(ARRAY_SIZE(funcs) == num_insns, "func table must match insn enum");
```

## Bug 2：mmu_alloc() 未固定映射地址导致 TO_HOST 体系失效（ASan 指向 machine_setup/mmu_write）

### 现象

启用 ASan 后，崩溃稳定出现在启动阶段 `machine_setup()` 写 argv 的 `mmu_write()`：

```
AddressSanitizer: SEGV in __asan_memcpy
  mmu_write -> machine_setup -> main
```

### 根因

`TO_HOST(addr)` 依赖“Guest 地址 + 固定偏移”的映射约定。

`mmu_load_segment()` 对 ELF 段使用了 `MAP_FIXED`，保证映射在期望虚拟地址。

但 `mmu_alloc()` 扩展匿名页时使用了：

```
MAP_PRIVATE | MAP_ANONYMOUS
```

这只把 `mmu->host_alloc` 当作 hint，内核可能返回不同地址，导致：

- 我们认为 Guest 的 heap/stack 在 `GUEST_MEMORY_OFFSET + xxx`；
- 实际 mmap 出来的内存不在这个地址；
- 之后 `mmu_write()` 对 `TO_HOST(guest_addr)` 发生写入未映射地址，直接 SIGSEGV。

### 修复

文件：`/home/jason/prj_c/riscv/riscvemu/src/mmu.c`

- `mmu_alloc()` 扩展时增加 `MAP_FIXED`，保证匿名映射地址与 `host_alloc` 一致。
- 修正缩减时 `munmap` 的起始地址，按“对齐后的 alloc 对应 host 地址”去 unmap。

## Bug 3：运行过程中 pc 变为 0，继续取指会段错误（原版 rvemu 也会在类似场景报错）

### 现象

在某些路径下，Guest `pc` 变成 0（常见于 `ret` 到 `ra=0` 或栈被破坏导致返回地址为 0）。

如果继续执行：

```
u32 data = *(u32 *)TO_HOST(state->pc);
```

会去读 `TO_HOST(0)` 处的指令，触发段错误。

### 修复（兜底）

文件：`riscvemu/src/interpret.c`

当 `state->pc == 0` 时，设置 `exit_reason = halt` 并退出解释执行循环，避免继续非法取指。

同时扩展退出原因并在调度层接住：

- `inc/riscvemu.h`：新增 `halt`
- `src/machine.c`：允许返回 `halt`
- `src/riscvemu.c`：主循环遇到 `halt` 时 break

这一步是“安全收敛/防崩溃兜底”，不是最终的语义正确性保证；后续真正要做的是定位为什么 `pc` 会变 0。

## Bug 4：为什么 qemu 有输出，而 riscvemu 没输出（关键：sys_fstat 写错结构体导致栈被覆盖）

### 现象

`qemu-riscv64-static testrv` 正常打印：

```
Hello, RISC-V Emulator!
```

而 riscvemu 在某阶段无输出（或只跑到 halt 退出）。

通过 syscall trace 观察到，Guest 在打印之前会先做：

- `ecall n=80`（SYS_fstat）
- `ecall n=214`（SYS_brk）
- `ecall n=64`（SYS_write）才真正输出

如果 `fstat` 失败或写坏内存，后续 `write` 可能根本到不了。

### 根因

原先 `sys_fstat` 直接调用：

```c
return fstat(fd, (struct stat *)TO_HOST(addr));
```

这里存在严重问题：`struct stat` 的布局是“主机架构相关”的（当前 host 为 x86_64），但 Guest 期望的是 RISC-V/Linux 目标 ABI 的 `struct stat` 布局。

把 x86_64 的 `struct stat` 原样写进 Guest 内存，会造成：

- Guest 读取 `st_mode` 等字段得到错误数据；
- 更严重的是大小/布局不一致会覆盖 Guest 栈上其它内容（例如返回地址、局部变量等），导致控制流异常；
- 最终表现为“没有输出/pc 归零/halt”。

### 修复

文件：`riscvemu/src/syscall.c`

- 改为：Host 侧 `fstat()` 到本地变量 `struct stat st`
- 再把需要字段转换拷贝到一个明确布局的 `riscv_stat_t`（128 bytes，asm-generic 风格）并 `mmu_write()` 写入 Guest 指针地址。
- 避免使用 `st_atime/st_mtime/st_ctime` 这些在 glibc 中可能被宏替换的字段名，改成 `st_atime_sec` 等。

修复后，`./riscvemu playground/testrv` 可以正常触发 `SYS_write` 并打印字符串。

## 额外：辅助排查能力（syscall trace）

文件：`riscvemu/src/riscvemu.c`

新增环境变量开关：

```bash
RISCVEMU_TRACE_SYSCALL=1 ./riscvemu playground/testrv
```

会在 stderr 输出 ecall 的编号与 a0-a3 参数，便于快速判断程序卡在哪个 syscall。

## 后续建议

1. `halt` 当前是“防崩兜底”。如果要进一步提升语义正确性，可以在 `pc==0` 时打印一次关键信息（pc/ra/sp/最后一条指令），但这会引入额外输出行为，需要按你的调试需求再决定。
2. syscall 结构体兼容是大坑：`stat/timeval/rlimit/sigaction/...` 都可能有类似的 ABI 差异。后续如果你开始跑更复杂的 ELF/LibC 程序，建议逐步补齐这些结构体的 guest layout 映射。
